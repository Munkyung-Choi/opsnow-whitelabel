# Decision Journal: 2026-04-22 (2차)

> 2차 갱신 — 1차 저널에 WL-150(High) / WL-152(Low) / WL-153(Critical) 완료 + WL-154 티켓 발행 + DEBT-007 Issue 1 상환 서사 통합.

## Decisions

- **§1 관심사의 분리 3.5 → 4.0 경로는 "자동화 거버넌스 확립"이다.** `WL-147` + `WL-148`로 God Component 해체 후 `WL-149` ESLint `max-lines/complexity`로 **재발 방지**. Ratchet 전략(기존 부채 현 수치 고정 + 개선 시 하향). "한 번 잘 정리했다"가 아닌 "시스템이 스스로 경계 유지"가 4.0의 본질이라는 인식 전환.
- **외부 AI 설계 조언은 반드시 코드 대조 후 수용한다.** 세션 전체에서 `/aisend` 스킬 교차검증 **12회 라운드** 수행. 오진 감지 누적: `WL-146` Server Action 런타임 위치·GET DB 호출 (2건), `WL-147` 이미지 업로드 통합 (1건), `WL-148` Zod derive (1건), `WL-150` Jira 원안 CHECK 서브쿼리·RPC `||` Trap (2건), `WL-150` 외부 AI "로컬 Impl 즉시" 프로토콜 위반 (1건), `WL-153` `CREATE VIEW ... SET search_path` 함수 전용 문법 오용 (1건). **외부 AI의 이론 레벨 조언은 실행 가능성(Feasibility) 실측 검증 없이 수용 금지.**
- **`WL-150`/`WL-151` 분리 진행으로 원자성 보존.** 외부 AI가 "통합 보안 강화" 옵션을 제시했으나 트랙 등급(High vs Low)과 롤백 단위가 달라 분리가 맞음. 순서는 사이클 짧은 Low(`WL-151`) 먼저.
- **WL-150 Critical 자동 승격 회피 — Pre-flight 실증의 힘.** Cloud `SELECT ... WHERE NOT is_valid_features(features)` 실행 0건 → 기존 partners 전원 `features='{}'` 실증 → backfill 불필요 → High 트랙 유지 확정. "불확실하면 Critical" Fail-safe 원칙의 반대 경로 — **실측으로 불확실성을 제거해 하위 트랙 유지**.
- **DEBT-007 부분 상환 전략 — 하나의 부채를 여러 티켓으로 쪼개 운영.** Issue 1(leads_masked_view)과 Issue 2(anon INSERT)는 성격(뷰 vs RLS+앱 코드)과 작업 규모가 달라 분리. `WL-153`(Critical)으로 Issue 1 당일 해소, `WL-154`(Critical, 내일) Issue 2 예정. DEBT-007 자체는 "부분 상환" 상태로 `tech-debt.md`에 서술.
- **WL-152 메타 프로세스 개선 — 훅과 운영 규칙의 충돌을 발견·해소.** WL-150 커밋 시 `pre-commit` Drift Precheck가 스테이징된 신규 마이그레이션을 Cloud 미적용 drift로 오판 → `--no-verify` 우회 필요 발생. 즉시 별도 티켓(WL-152) 분리 후 당일 해결, WL-153 커밋에서 개선된 훅의 실전 증명(`✅ Drift Precheck 통과`). 방어선이 정상 작업을 방해할 때는 방어선 자체를 교정해야 한다는 원칙.
- **Layered Defense 확립 — WL-150 / WL-153 공통 패턴.** WL-150: Layer 1 RPC allowlist + pruning / Layer 2 DB CHECK 제약 / Layer 3 ESLint 가드(WL-151). WL-153: SECURITY DEFINER 뷰 + schema-qualified 참조 + WHERE EXISTS(master_admin) access control. 단일 방어선 실패 시 다른 레이어가 여전히 신호를 주도록 설계.
- **인스턴스 기반 점진적 정제 — "자가 치유" 수사의 구체화.** WL-150 RPC `jsonb_build_object` 재구성 전략은 **전역 자동 청소가 아니라 RPC 호출된 파트너 row만 allowlist 3 key로 재작성**. 운영 트래픽이 자연스럽게 legacy key를 정제함. 외부 AI의 "자가 치유" 수사를 내부 문서에서는 "인스턴스 기반 점진적 정제"로 구체화.

## Rejected Paths

- **`WL-148` Zod ↔ FieldDef 타입 결합 (Candidate D)** — Template literal types로 `K in keyof Schema` 강제 가능했으나 i18n prefix 관계로 단순 적용 불가. 14 Unit test 런타임 가드로 충분 → ROI 부족.
- **Partners 폼 FieldDef 패턴 전파 (Candidate B)** — `partners/[id]/page.tsx` 96줄 단순 폼에 선언적 패턴 이식은 과잉 설계. 닭 잡는 데 소 잡는 칼.
- **`usePreviewBridge` 범용 위치 승격 (Candidate C)** — 2번째 에디터 등장 전엔 premature abstraction. YAGNI.
- **`WL-146` 이미지 업로드 로직 훅 통합** — `SiteBuilderForm`과 `ContentEditorForm`이 별개 페이지임을 코드 대조로 확인. 외부 AI가 "SiteBuilder니까 묶여있겠지"라고 가정한 오진.
- **`WL-146` `beforeunload` 경고 도입** — 브라우저 호환성 이슈 + E2E flaky 우려. UI 내 dirty dot(`•`)만 유지.
- **`pg_jsonschema` extension 도입 (WL-150 설계)** — Supabase 인프라 변경 최소화. 기본 PostgreSQL 함수(`jsonb_typeof`, `jsonb_object_keys`)로 대체.
- **jsdom 컴포넌트 단위 테스트 인프라 즉시 도입** — `WL-148` Test Contract 조정 시 대두. 별도 티켓 영역으로 분리, 현재 E2E로 대체.
- **WL-150 Jira 원안 CHECK 서브쿼리 구조** — `CHECK (... AND (SELECT bool_and(...) FROM jsonb_object_keys(features) AS key))`는 PostgreSQL "CHECK expressions cannot contain subqueries" 규칙 위반. **IMMUTABLE 함수로 우회**하여 함수 내부에 `jsonb_each` 서브쿼리 격리 + search_path 고정으로 injection 방어.
- **WL-150 RPC `||` 병합 유지** — Auditor 취약점 #3: legacy key 보유 행에서 정상 key 토글 시 CHECK 위반으로 정상 작업까지 영구 차단 Trap 위험. **`jsonb_build_object` 재구성으로 전환**.
- **WL-150 훅 로직 수정을 WL-150 커밋에 포함** — 원자성 훼손. WL-152 별도 후속 티켓으로 분리, WL-150은 `--no-verify` 우회 + 사유 커밋 메시지 기록으로 마무리.
- **WL-150 types/supabase.ts `features: Json` → `FeaturesMap` 수동 override** — auto-gen `gen:types` 재실행 시 원복 위험. DEBT-012 구조 개선(DB `allowed_feature_keys` 참조 테이블) 선행 후 함께 해결. 지금 단독 도입은 부채를 키움.
- **WL-153 외부 AI `CREATE VIEW ... SET search_path = pg_catalog, public` 구문** — PostgreSQL VIEW는 `SET search_path` 옵션 미지원(함수 전용 문법). **schema-qualified 참조(`public.leads`, `public.profiles`)로 대체**하여 OID 고정 수준 방어.
- **WL-153 `leads` 테이블에 master_admin 전용 SELECT RLS 정책 추가** (대안 Option B) — master_admin이 `leads` 원본에 직접 접근 가능해져 마스킹 의도 훼손. SECURITY DEFINER 뷰 방식이 원칙에 더 부합.
- **WL-150 B-2(anon INSERT) + B-1을 하나의 티켓으로 통합** — 앱 코드 변경 + E2E 확인 동반 여부가 달라 성격 분리 필요. B-1은 SQL 1줄, B-2는 Server Action 변경 + ContactForm E2E 검증. WL-153/WL-154로 분리.
- **외부 AI 3차 라운드 "로컬 Impl 즉시 시작" 권유** — CLAUDE.md High 트랙 비동기 Human Check 규정("Impl은 승인 후") 위반. Q1 Pre-flight 실측까지는 대기, 대기 중 Test Contract까지만 진행.
