## 📝 Core Logic

1. **정합성 모델 전환**: `CHECK (default_locale IN ('ko','en'))` 화이트리스트 → `CHECK (default_locale = ANY(published_locales))` 관계적 정합성으로 교체. 사업 계획(ko/en/ja/zh + da 등 확장)을 DB 레벨에서 지원.
2. **Option B-2 채택**: default_locale은 반드시 해당 파트너의 published_locales array 원소여야 한다는 불변. 빈 배열이나 고아(orphan) default를 DB 레벨에서 원천 차단.
3. **원자적 실행**: `BEGIN; ... COMMIT;` 명시적 트랜잭션으로 DROP→ADD 사이의 방어 구간 없음.

## 🔄 롤백 정책 (Option A — Forward-only)

본 마이그레이션은 **확장(관용 확대) 방향**이므로 화이트리스트 복원을 롤백으로 간주하지 않는다.

- **긴급 복구 시**: `ALTER TABLE public.partners DROP CONSTRAINT IF EXISTS partners_default_locale_in_published;` — 제약만 제거, 재부과 없음.
- **재부과 금지 이유**: ja/zh 파트너가 이미 등록된 이후 `CHECK (default_locale IN ('ko','en'))` 복원은 기존 데이터 위반으로 실패함.
- **대안 없는 순방향 설계**: 비즈니스가 다국어 지원 방향으로 이미 진행 중이므로, 복구는 "앞으로" 나아가는 것으로 정의한다.
