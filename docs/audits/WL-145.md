## Auditor Digest

### Core Logic
파트너별 sliding window rate limit (기본 50 req/60s) 게이트를 `src/proxy.ts`의 3개 경로에 삽입.
- (1) DEV fallback `resolvePartnerBySubdomain` 성공 후 — Vercel Preview URL 공격면 차단
- (2) 마케팅 host — `resolvePartnerFromHost` **전** host 기반 선행 게이트 (Partner Enumeration 방어) + 성공 후 `partner.id` 기반 메인 게이트
- Upstash Redis + `ephemeralCache` + `analytics: false`로 무료 티어 10k/day 보존. Lazy singleton + env 검증으로 Fail-open 보장.

### Potential Risks
1. **Upstash quota 소진**: `analytics: false`로 가시성 없음 → 10k/day 한도 도달 시 Fail-open 무한 통과. **DEBT-010**으로 별도 관리 (quota monitoring Webhook).
2. **Admin host 무방비**: `admin-whitelabel.opsnow.com`은 `resolvePartnerFromHost` 경로를 타지 않아 rate limit 미적용 → credential stuffing 통로. **WL-146 신규 티켓**으로 IP 기반 방어 분리.
3. **Public API 경로 무방비**: `/api/set-locale`·`/api/admin/*` Route Handler는 proxy matcher에서 `api` 제외됨 → rate limit 우회 가능. **DEBT-009**로 별도 관리.

### Questions for Reviewer
1. **Fail-open 정책의 DDoS 악용**: 공격자가 Upstash 한도를 의도적 소진 후 업무 시간대 무제한 통과 창 형성 가능. 2단 in-memory 폴백 도입 여부?
2. **격리 키 선택**: `partner.id` vs `host` vs `host+ip` — Edge 다중 region 동시 요청 시 sliding window 일관성 보장 수준?
3. **Dev/Prod 설정 분리**: E2E 병렬 실행 시 50/60s가 너무 타이트해 정상 플로우 차단 — env 기반 override가 올바른 접근인가?

### Resolution

**Q1 — Fail-open DDoS 악용 (Human Check 결정 2)**:
- **결정**: **[A] 완전 Fail-open 유지**. 가용성(매출) 우선. 2단 폴백은 Edge Isolate 격리로 정확도 낮음 + 복잡도만 증가.
- **보완**: `console.warn('[RateLimit] Fail-open triggered')` 로깅으로 가시성 확보. DEBT-010으로 quota monitoring 체계 구축 예정.
- **근거**: `src/lib/rate-limit.ts:46-48`, `src/proxy.ts:285-304` checkRateLimit catch 블록.

**Q2 — 격리 키 선택**:
- **결정**: **`partner.id` (UUID) 2차 + `host:cleanHost` 1차 2단 구조**.
- **근거**: `partner.id`는 정규화된 단일 진실 원천 (custom_domain 변경에도 불변). Host 기반 선행 게이트는 Partner Enumeration 공격을 DB 조회 전에 차단 — Supabase `partners` 테이블 부하 원천 차단.
- **트레이드오프**: 다중 Edge region 간 sliding window 일관성은 근사 보장 (Upstash 공식 ~95% 정확). ephemeralCache가 Isolate별 독립이므로 전역 정확도는 이에 의존하지 않음.

**Q3 — Dev/Prod 설정 분리**:
- **결정**: **환경별 기본값 + env override**. `NODE_ENV !== 'production'` → 500/60s, 'production' → 50/60s. `RATE_LIMIT_REQUESTS`/`RATE_LIMIT_WINDOW` 명시 override.
- **근거**: Playwright 4 worker 병렬 실행 시 partner-a에 대해 50 req/60s 한도 초과로 정상 테스트가 차단됨 (Bucket C). Dev 기본값을 500으로 완화하여 regression 0 확인.
- **검증 결과**: `npx playwright test` 206 pass / 0 regression (이전 21 failed → 0).
