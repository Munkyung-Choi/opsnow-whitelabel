## Auditor Digest

### Core Logic

DEBT-007 Issue 1 해소 — `leads_masked_view`의 **security_invoker=true**(Gemini 감사 수정의 역효과)로 master_admin이 마스킹 뷰 조회 시 0건 반환되던 갭을 해소. **SECURITY DEFINER(invoker=false)** 전환으로 뷰 소유자(postgres superuser) 권한 실행 → `leads` 접근 가능 → 뷰 내부 `WHERE EXISTS (master_admin)` 절이 access control 담당. 추가로 `public.leads`, `public.profiles` schema-qualified 참조로 search_path hijacking 방어.

### Potential Risks (본 세션에서 실증 검증)

1. **[해소] Gemini 감사 수정의 역효과**: `ALTER VIEW ... SET (security_invoker = true)` 설정이 master_admin의 `leads` SELECT 정책 부재와 조합되어 `FROM leads`에서 RLS 0건 필터 → 뷰 내부 `WHERE EXISTS (master_admin)` 필터 도달 전 차단됨. → security_invoker=false로 복원.
2. **[방어 적용] search_path hijacking**: SECURITY DEFINER 뷰는 superuser 권한으로 실행되므로 `search_path` 조작 공격 노출 가능. → `public.leads`, `public.profiles` 명시로 OID 고정 수준 방어.
3. **[방어 유지] access control 유실 방지**: `WHERE EXISTS (profiles.role = 'master_admin')` 절이 partner_admin / anon 세션에서 여전히 0건 반환 보장. RLS 통합 테스트로 실증.
4. **[외부 AI 오진 1건 교정]**: 외부 AI가 `CREATE VIEW ... SET search_path = pg_catalog, public` 구문을 제안했으나 PostgreSQL 뷰에 `SET search_path` 옵션이 없음(함수 전용 문법). schema-qualified 참조 방식으로 대체.

### Questions for Reviewer

1. **Q1 — 뷰 소유자 권한 사전 검증**: `viewowner = postgres` 확인이 SECURITY DEFINER 전환의 전제 조건이었음. (보안/아키텍처)
2. **Q2 — 외부 AI가 제안한 `CREATE VIEW ... SET search_path` 구문의 실행 가능성**: PostgreSQL 문법 규칙 검증 필요. (문법 정합성)
3. **Q3 — access control 경로 유지 증명**: partner_admin / anon 세션에서도 기존대로 0건 반환되는지? (설계 연속성)

### Resolution

**Q1 — 뷰 소유자 권한 사전 검증 완료**:
- **결정**: 2026-04-22 Cloud SQL Editor `SELECT viewowner FROM pg_views WHERE viewname = 'leads_masked_view'` 실행 → `postgres` 확인.
- **근거**: Supabase 기본 superuser로 소유되어 있어 SECURITY DEFINER 방식에서 `leads` 테이블 접근 권한 보장됨.

**Q2 — 외부 AI 제안 SQL 문법 오류 교정 완료**:
- **결정**: `CREATE VIEW ... SET search_path = pg_catalog, public`은 **PostgreSQL 뷰에서 지원되지 않는 문법**(함수 전용). 실행 시 `syntax error at or near "SET"` 발생.
- **역제안**: 뷰 정의 내 모든 테이블 참조를 schema-qualified(`public.leads`, `public.profiles`)로 명시 → OID 고정으로 search_path hijacking 방어 동일 효과 달성.

**Q3 — access control 경로 유지 증명 완료**:
- **결정**: 적용 후 `rls-isolation.test.ts` 3개 테스트 모두 통과:
  1. `[design-intent]` master_admin도 `leads` 직접 SELECT 불가 — access control 유지
  2. `[access-control]` partner_admin은 `leads_masked_view` 조회 불가 (0건) — access control 유지
  3. `[intended-design / WL-153]` master_admin은 마스킹된 이메일(`*******@company-e.test`) 조회 가능 — 설계 의도 복원
- **근거**: `rls-isolation.test.ts:541-568` (WL-153 수정 후)

### 외부 AI 오진 기록 (Negative Knowledge Base)

- **오진 내용**: `CREATE VIEW ... SET search_path = pg_catalog, public` 구문 제안
- **오진 이유**: PostgreSQL에서 `SET search_path` 옵션은 `CREATE FUNCTION` / `CREATE PROCEDURE`에서만 지원. VIEW는 지원하지 않음.
- **올바른 대안**: 뷰 정의 내부에서 모든 테이블 참조를 schema-qualified로 명시 (`public.table_name`). OID 고정 수준 방어로 search_path hijacking 동일하게 차단.
- **학습**: 외부 AI가 "보안 표준"을 이론 레벨로 전달할 때 **실행 가능성(Feasibility)을 반드시 실측으로 증명**할 것. 문법 오류는 적용 시 마이그레이션 전체 실패로 이어짐.
