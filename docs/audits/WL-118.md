## 📝 Auditor Digest

### Core Logic
Vitest 기반 RLS 격리 자동 검증 스위트. 5개 테이블(contents/leads/partners/domain_requests/system_logs)에 대해 Triangular Assertion(positive-control/negative-control/target) 패턴으로 파트너 간 데이터 격리를 검증. `setSession()` 전용(헤더 주입 금지), Shadow Data Preflight로 "RLS 차단 vs 데이터 부재" 혼동을 원천 차단.

### Identified Risks

**R-LF-1 (Logic Flaw)**: 0건 반환만으로는 RLS 정책 차단인지 데이터가 실제로 없는 것인지 구분 불가.
→ **해결**: Triangular Assertion 도입 — positive-control(master_admin 접근 가능)으로 데이터 존재를 먼저 증명한 뒤 target(partner-a)의 0건 반환이 격리를 의미함을 확정.

**R-DI-2 (Data Integrity)**: 테스트 중 `partners` 테이블에 신규 INSERT 금지.
→ `trg_init_partner_defaults` 트리거가 발화하면 `contents` 11개가 자동 생성되어 다른 테스트의 기대값(row count, id 조회) 붕괴. 기존 seed 4개 파트너만 사용.

**R-AU-1 (Auth Bypass Risk)**: 헤더 주입 방식(`Authorization: Bearer`) 사용 시 `auth.uid()` ≠ `JWT.sub` 불일치가 발생하고 `get_my_role()` NULL 반환 가능.
→ **해결**: `signInWithPassword()` → `setSession()` 단일화. 헤더 주입 fallback 코드 없음.
