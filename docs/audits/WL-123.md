## 📝 Auditor Digest

### Core Logic

`system_logs.partner_id UUID` 컬럼을 추가하여 파트너별 감사 로그 직접 필터링(NFR §5.3)을 구현한다. `withAdminAction`의 자동 주입은 `PARTNER_SCOPED_TABLES` 화이트리스트에 한정되며(Default Deny), master_admin 누락 시 런타임 경고로 명시성을 강제한다. RLS 정책은 `partner_id OR on_behalf_of` 병행으로 레거시 호환을 유지하며 WL-122와 동일한 `BEGIN/COMMIT` 패턴으로 권한 공백을 원천 차단한다.

### Decisions (HIGH-Critical Pivot 결과)

| 결정 | 채택안 | 근거 |
|------|--------|------|
| R-LF-3: master 작업 로그 공개 | **B+ (화이트리스트 기반 선별 공개)** | WL-121의 "비공개" 의도 + NFR §5.3의 "추적성" 의도 양립. 파트너-스코프 테이블에만 국한 |
| R-DI-2: INDEX 전략 | **평문 `CREATE INDEX IF NOT EXISTS` 트랜잭션 내** | Shadow Migration 실행 가능성 + 프로덕션 2,400행 규모에서 Lock 체감 불가 |
| R-LF-1: master 직편집 partner_id 누락 | **명시적 주입 + 런타임 `console.warn`** | AI-Native 공정에서 명시성 = 안정성. URL 자동 감지는 fragile → 거부 |

### PARTNER_SCOPED_TABLES 화이트리스트 (Default Deny)

```typescript
const PARTNER_SCOPED_TABLES = [
  'partners', 'contents', 'partner_sections',
  'leads', 'site_visits', 'domain_requests',
] as const
```

**제외 테이블 근거**:
- `profiles` — 개별 유저 self-profile 수정까지 파트너 전체에 노출할 필요 없음
- `global_contents` — 전역 리소스, 파트너 소속 없음
- `system_logs` 자체 — 감사 로그의 로그는 master 전용

### resolvePartnerId 규칙 (단일 진입점)

우선순위 순:
1. `auditDetails.partner_id` 명시 → 그 값 (최우선)
2. `auditDetails.on_behalf_of` 존재 → impersonation 대상으로 복사
3. `user.role === 'partner_admin' && target_table ∈ PARTNER_SCOPED_TABLES` → `user.partner_id`
4. `user.role === 'master_admin' && target_table ∈ PARTNER_SCOPED_TABLES && target_id 존재 && partner_id/on_behalf_of 미명시` → `console.warn` + NULL 반환 (개발자에게 명시 누락 신호)
5. 그 외 → NULL (글로벌 작업)

### 데이터 분포 실측 (R-SM-1 증거)

**프로덕션 (AP Architect 공유, 2026-04-19 기준)**: 총 약 2,400행

**로컬 DB 실측 (2026-04-19)**:
```sql
SELECT (on_behalf_of IS NULL) AS oh_null, COUNT(*)
FROM system_logs GROUP BY 1;
-- → (0 rows)
```

**결과 해석**: 로컬 개발 환경은 Admin 기능을 사용한 이력이 없어 `system_logs` 테이블이 비어 있음. 따라서 로컬은 **backfill 대상이 존재하지 않는 무해 상태**이며, 실질적 backfill 검증은 프로덕션에서만 유의미하다.

**프로덕션 검증 계획 (문경 님 SQL Editor 실행)**:
```sql
SELECT (on_behalf_of IS NULL) AS oh_null,
       (partner_id IS NULL) AS pid_null,
       COUNT(*)
FROM system_logs
GROUP BY 1, 2;
```

| oh_null | pid_null | 예상 의미 | 전략 |
|---------|----------|----------|------|
| TRUE | TRUE | 일반 master_admin 작업 (대부분) | 변경 없음 — 영구 NULL |
| FALSE | TRUE | impersonation (backfill 대상) | Part A UPDATE로 `partner_id ← on_behalf_of` 복사 |
| FALSE | FALSE | 신규 master 직편집 or 재실행 | 변경 없음 (멱등성) |
| TRUE | FALSE | 이론적 불가능 케이스 | `UPDATE ... WHERE on_behalf_of IS NULL AND partner_id IS NOT NULL`로 감지 시 경고 |

**Part A 마이그레이션은 프로덕션에서 `WHERE on_behalf_of IS NOT NULL AND partner_id IS NULL` 조건으로 멱등 실행 안전**.

### 검증 계획

**Unit (Vitest)**: `resolvePartnerId`의 4규칙 × 경계 케이스
- Rule 1: 명시 우선 (다른 조건 무시)
- Rule 2: impersonation 복사
- Rule 3: partner_admin 스코프 테이블 + 비스코프 테이블
- Rule 4: master_admin 스코프 + 미명시 → warn + NULL
- Fallthrough: 글로벌 작업 → NULL

**E2E (Playwright)**: Triangular Assertion
- partner-a 유저가 자기 partner_id 로그만 SELECT 가능
- partner-b 로그는 조회 불가
- master 직편집 시 partner_id=partner-a 로그가 partner-a 유저에게 노출됨
- master 글로벌 작업(target_table ∉ whitelist) 로그는 partner-a 유저에게 비노출
