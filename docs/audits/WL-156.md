## Auditor Digest

### Core Logic

Next.js 16.x는 `src/proxy.ts`를 미들웨어 진입점으로 **직접 인식**(commit b381220 참조) — Next.js가 `proxy(request: NextRequest, event: NextFetchEvent)`를 호출. Rate Limit 초과(429) 발생 시 `checkRateLimit` 내부에서 `event.waitUntil(dispatchRateLimitAlert(ctx))`로 알림을 동기 등록한 뒤 429를 즉시 반환. `dispatchRateLimitAlert`는 Upstash Redis `SET NX + EX 300s` 원자 패턴으로 5분 suppression 후 Resend SDK로 이메일 발송. 3계층 방어(Redis 장애 침묵 / Resend 실패 catch / outer try-catch)로 메인 응답 영향 차단.

### Identified Risks

Auditor sub-agent 독립 검증 결과 4건 발견, 모두 Design v2에 반영:

1. **[HIGH] Context 주입 경로 미설계** — `checkRateLimit`이 429만 반환하고 호출부별 context 결정 방법 미명시. `/auth/*`(`proxy.ts:344-348`)와 Admin host(`proxy.ts:354-359`)가 동일 `ip:` key + 동일 limiter 공유 → 알림 혼동. **해결**: `checkRateLimit(key, limiter, { context, ip, event })` 시그니처로 명시적 주입 강제. 5곳 호출부가 각자 context 지정.
2. **[무효화 — 전제 오류 발견]** Auditor는 당초 "proxy.ts `config` 제거 후 잔존 방어선"을 HIGH로 지적했으나, **Implementation 중 Next.js 16.x가 proxy.ts를 미들웨어 진입점으로 인식한다는 사실이 실증**(curl `/__proxy_health` 200 + `x-middleware-rewrite` 헤더 확인 + commit b381220 히스토리)됨. `export const config`는 **제거 대상이 아닌 유지 필수**. `tests/lib/proxy/export-surface.test.ts`의 assertion 방향을 "config **export해야 함**"으로 반전. WL-156 scope 항목 4(middleware.ts 신규) 자체가 거짓 전제였으며, 실제 작업은 proxy() 시그니처 확장(event 수신) + 429 분기 waitUntil 주입으로 축소.
3. **[MEDIUM] Suppression key space 내 context 미포함** — `alert:ratelimit:${ctx.key}`가 감사 로그 혼동 유발. **해결**: `alert:ratelimit:${ctx.context}:${ctx.key}`로 네임스페이스에 context 포함.
4. **[MEDIUM] `event.waitUntil` 이중 호출 → Silent Loss** — outer + inner waitUntil 중첩 시 Cold start 직후 Isolate shutdown에서 inner no-op → 알림 손실 + 5분간 suppression 침묵. **해결**: `dispatchRateLimitAlert` 내부 중첩 waitUntil 제거, 호출부 1회만 등록 + 내부는 `await sendEmailAlert(...).catch(...)` 직렬 체인.

### Resolution

- SET NX + EX 원자 suppression 패턴 유지 (GET-then-SET 역전 기각)
- Fail-Safe 침묵 (Redis 장애 시 알림 포기, 이메일 폭주 방지)
- 3계층 방어선 유지 (outer try-catch / `.catch()` / Redis lazy null)
- AlertContext 평면 string union 유지 (YAGNI — 확장 실요구 시 마이그레이션)
- Upstash Redis fetch REST stateless 전제 (재연결 불필요)
- Pre-flight smoke check 단계에서 resend Edge 호환성 실증 조건부 승인
- WL-157(E2E baseURL)·WL-158(env Zod)·WL-159(observability) WL-156 완료 후 발행

### 전제 교정 기록 (2026-04-23 Implementation 단계)

**Critical Discovery**: WL-156 티켓 설명의 "middleware.ts 부재로 proxy.ts가 미들웨어 인식 안됨, rate limiting 비활성" 전제가 **실측 대조 결과 거짓**으로 확인됨.

- **증거 1**: Next.js 16.x 공식 동작 — `src/proxy.ts`가 미들웨어 파일로 인식됨 (git commit b381220 "[WL-46] fix: middleware.ts 제거" 이미 명시)
- **증거 2**: Apr 22 Playwright 결과(`playwright-results.json`) — 209 expected / 0 unexpected 통과 → 미들웨어 활성 상태에서만 가능한 수치
- **증거 3**: Impl 후 실측 curl — `/__proxy_health` 200 + `x-middleware-rewrite: /{uuid}/ko` 헤더 확인

**오진 원인**: journal 2026-04-22 L19 "`.next/dev/server/middleware-manifest.json`의 `middleware: {}`로 실증"은 Next.js 16의 manifest 위치/구조 변경을 오해한 것. 실제 매니페스트는 다른 곳에 존재하거나 dev 모드에서 생성 경로 변경됨.

**피벗 조치**:
