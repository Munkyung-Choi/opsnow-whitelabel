## Auditor Digest

### Core Logic
Admin host 경로에 IP 기반 sliding window rate limit 게이트를 `src/proxy.ts`에 2단 배치.
- **Gate 1 (`/auth/*` 경로)**: `pathname.startsWith('/auth')` 조기 통과 직전. Server Component `auth.getUser()` + Server Action `signInWithPassword` 모두 우리 서버 경유 Supabase Auth API 호출 발생 → 공격면.
- **Gate 2 (`isAdminHost(host)` 분기)**: `supabase.auth.getUser()` 세션 검증 **이전**. 페이지 접근 자체로 Supabase Auth API 호출이 증폭되지 않도록 원천 차단.
- Upstash Redis 동일 인스턴스, **독립 ephemeralCache + `_initAttempted` 가드** 분리. prefix `rl_admin`으로 파트너 limiter(`rl_partner`)와 충돌 방지.

### Potential Risks
1. **공유 IP 환경 (오피스 NAT)**: OpsNow 사무실 등 다수 운영자가 한 IP 공유 시 30 req/60s(prod) 한도 공유.
   - **완화**: `RATE_LIMIT_ADMIN_REQUESTS` env로 즉시 상향 조정 가능. 페이지 전환 자체가 matcher에서 정적 자원(`_next/static`·`images/`·`api`) 제외되어 1 req/페이지 수준 → 30/60s는 페이지 기준 매우 여유.
2. **`/auth/*` Gate 1 우회 경로 없음 확인**: `/auth/callback` 등 `/auth/*` 하위 경로도 동일 게이트에 포함되어 있음. Server Action은 form URL(`/auth/login`)로 POST되므로 Gate 1에 걸림.
3. **`/api/auth/*` Route Handler 미방어**: `src/proxy.ts:438` matcher에서 `api` 제외 → WL-146 범위 밖. **DEBT-009**로 별도 관리 (WL-145에서 이미 등록).

### Questions for Reviewer

1. **`/auth/login` GET은 실제로 Supabase Auth API를 호출하는가?** — 외부 AI 1차 승인 당시 "GET은 DB 호출 없음"으로 제외하기로 했으나 코드 대조 결과 `src/app/auth/login/page.tsx:12-21`에서 `supabase.auth.getUser()` 호출 확인. Rate limit 대상 포함 여부?
2. **격리 키 선택**: `ip:<client-ip>` vs `ip:<client-ip>:<user-agent>` — User-Agent 혼합 시 봇의 UA 로테이션 회피 가능성?
3. **Admin과 Partner ephemeralCache 공유 vs 분리** — 공유 시 eviction 상호 간섭, 분리 시 메모리 사용량 2배.

### Resolution

**Q1 — `/auth/login` GET rate limit 포함 결정**:
- **결정**: **Gate 1 (`/auth/*`) 진입 직전 IP rate limit 적용**. 외부 AI의 초기 "GET 제외" 주장은 오진 — Server Component에서 `auth.getUser()` 실행이 실제 Supabase API 호출로 이어짐.
- **근거**: `src/app/auth/login/page.tsx:14-17` `const supabase = await createSessionClient(); const { data: { user } } = await supabase.auth.getUser()` — 매 GET마다 Supabase Auth 호출 확인.
- **트레이드오프**: 정상 운영자의 로그인 페이지 단순 접근이 30 req/60s 한도 안에서 동작. 봇의 탐색 남용은 차단.

**Q2 — 격리 키 선택**:
- **결정**: **`ip:<client-ip>` 단순 키**. User-Agent 혼합 미채택.
- **근거**: 봇이 UA를 로테이션하는 속도(초당 수십)보다 IP 로테이션이 훨씬 비용 큼 (proxy pool 필요). IP를 단일 키로 쓰면 공격자 비용이 최대. UA 혼합은 선량한 사용자의 브라우저 업데이트 시에도 버킷 분리를 유발하여 정상 사용성 저해.
- **보완**: `x-vercel-forwarded-for` 우선 → `x-forwarded-for` 첫 IP fallback → `unknown` fallback. IPv6는 콜론 포함되므로 쉼표 분리 전용.

**Q3 — ephemeralCache 분리**:
- **결정**: **분리**. Partner와 Admin이 각각 독립된 `Map<string, number>` 보유.
- **근거**: 파트너 트래픽(대량) 유입 시 FIFO eviction이 admin key까지 밀어내면 Upstash 호출 증가 + 무료 티어 10k/day 소진 가속. 메모리 사용량 증가는 Edge Worker 수명 한정이라 미미.
