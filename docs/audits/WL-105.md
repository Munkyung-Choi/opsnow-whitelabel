### Identified Risks

**Risk A — `contents_public_anon_read` 에 partner_id 필터 없음**
- `USING (is_published = true)` 만 검사 — anon이 Supabase 직접 쿼리 시 전 파트너 마케팅 콘텐츠 열람 가능
- 마케팅 콘텐츠는 공개 성격이므로 보안 위협 아님. 앱 레이어 WHERE partner_id 필터로 충분
- **결정**: 수정 불필요. RLS에 partner_id 필터 추가 시 anon 요청마다 partners JOIN 오버헤드 발생 → 현 설계 유지

**Risk B — `partner_sections` anon 정책도 partner_id 미필터**
- `USING (is_visible = true)` 만 검사. Risk A와 동일 근거로 수정 불필요

**Risk C — SQL Editor `SET LOCAL ROLE authenticated` 시뮬레이션 한계**
- SQL Editor는 postgres(service_role)로 실행. `SET LOCAL ROLE authenticated` 은 role 전환이지만 실제 JWT claims 처리는 PostgREST 레이어임
- 완전한 RLS 격리 검증은 실제 앱(E2E) 또는 Supabase CLI `supabase test db`에서 수행
- **결정**: SQL Editor 검증은 "best-effort" 수준. E2E M-1-2, M-1-4, M-1-5 테스트가 앱 레이어 격리를 커버
