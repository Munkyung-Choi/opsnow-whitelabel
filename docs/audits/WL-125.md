## Auditor Digest

### Core Logic

파트너 자산(로고·파비콘) 업로드·서빙 인프라. Supabase Storage 버킷을 **2개로 분리**(`partner-logos` 2MB / `partner-favicons` 512KB)하여 DB 레벨에서 타입별 크기 제약을 강제하고, `storage.objects` RLS 4개(SELECT public / INSERT·UPDATE·DELETE 테넌트 격리)로 접근을 통제한다. INSERT/UPDATE `WITH CHECK`에 경로 정규식(`^{uuid}/(logo|favicon)\.(png|jpg|jpeg|webp|ico)$`)을 강제하여 master_admin도 예외 없이 규약을 따른다. 헬퍼는 session-based 클라이언트 전용으로 service_role 의존을 제거했다.

### Potential Risks

1. **MIME/확장자 SSOT 드리프트** — 마이그레이션 `allowed_mime_types`와 `src/lib/storage/partner-asset.schema.ts`가 두 곳에서 정의된다. 둘 중 한쪽만 변경되면 Zod 통과 후 Storage 거부 또는 그 반대가 발생. 방어책: 마이그레이션 헤더 주석에 "schema.ts와 동기 필수"를 명시했으나 자동 강제는 없음. 향후 schema.ts 상수를 마이그레이션 생성 시 참조하는 codegen 고려 대상 (tech-debt).
2. **로컬 shadow migration의 Supabase Storage 의존** — `npx supabase migration up`이 로컬 Docker의 Storage 컨테이너 기동 여부에 따라 실패 가능. 본 티켓은 성공 확인. Windows Docker Desktop 환경에서 storage 컨테이너 미기동 시 향후 개발자가 재현 불가 위험.
3. **버킷 정책 소급 미적용** — `ON CONFLICT DO UPDATE`는 버킷 설정만 갱신. 기존 저장된 객체는 grandfather되어 이전 정책 그대로 서빙. 본 티켓은 신규 버킷이므로 해당 없음. 향후 정책 변경 시 별도 cleanup 마이그레이션 필요.

### Questions for Reviewer

1. **보안** — UPDATE 정책에 `USING`과 `WITH CHECK` 모두 명시하여 rename 공격을 차단했다. 이 방어가 충분한가, 혹은 Supabase Storage의 `move()` API가 `UPDATE` 외 경로로도 파일 이동을 허용하는가?
2. **확장성** — 파트너당 최대 2개 파일만 저장하므로 네임스페이스 충돌 없음. 향후 hero_image, og_image 등 자산 종류가 늘어나면 버킷을 추가 분리할지, 단일 버킷 + 타입별 프리픽스로 전환할지 결정 필요.
3. **대안 설계** — SVG XSS 방어를 위해 `image/svg+xml`을 완전히 제외했다. 로고 렌더링 품질에 대한 디자인팀 요구가 생기면 DOMPurify 서버사이드 sanitize를 도입한 별도 티켓(예: WL-xxx)으로 재검토.

### Resolution

**Q1 (보안)**: Supabase `storage.from().move()` 및 `copy()`는 내부적으로 `UPDATE storage.objects SET name = ...`로 구현되므로 본 `UPDATE` RLS 정책이 커버한다. `WITH CHECK` 명시로 rename 공격은 차단됨. 직접 DELETE + INSERT 조합도 각 정책에서 경로 격리로 차단. **충분**.

**Q2 (확장성)**: 현 MVP 스코프(로고·파비콘 2종)에서 버킷 2개 분리가 최선. 자산 종류 3종 이상으로 확장 시 단일 버킷 + type prefix 패턴으로 리팩터 제안. 별도 티켓으로 tech-debt 등록 대상.

**Q3 (SVG)**: 현 결정은 **영구적 배제 아님**. Confluence 4.4.3에 "SVG는 DOMPurify sanitize를 경유하는 별도 파이프라인으로만 허용" 메모 추가 여지. 본 티켓 스코프 밖.
