## Auditor Digest

### Core Logic

DEBT-007 Issue 2 해소 — `leads_public_insert` RLS 정책을 물리적으로 `DROP`하고 marketing 리드 수집의 유일한 쓰기 경로를 `submitLead` Server Action + `supabaseAdmin`(service_role)로 수렴. 신뢰 경계는 `resolvePartnerIdFromHost(host)` 단일 함수. Auditor MEDIUM을 수용해 `leadSchema.strict()` + INSERT payload 순서 고정(`{ ...parsed.data, partner_id, status }`)으로 Zod 레이어와 JS 레이어 이중 방어를 구성.

### Potential Risks (본 세션에서 실증 검증)

1. **[방어 적용] 필드 오염 (Field Pollution)**: `leadSchema`가 strict 없음 + INSERT payload에 `partner_id: partnerId` 선두 → 미래 리팩터(예: `.passthrough()` 도입 또는 순서 뒤집기)에서 spread override 취약점. → `.strict()` 추가 + 페이로드 순서 반전으로 이중 봉쇄.
2. **[방어 적용] 배포 순서 역전 시 가용성 장애**: 코드 배포 전에 Cloud SQL Editor로 DROP POLICY 먼저 실행 시 기존 prod anon 경로 전면 RLS violation. → Runbook에 "Vercel green 확인 후에만 SQL Editor 실행" 명시 + 롤백 SQL 사전 준비.
3. **[방어 적용] 거버넌스 침식**: marketing Server Action에서 service_role 첫 선례화로 후속 확산 리스크. → CLAUDE.md §4에 "submitLead 1건 한정 + 신규 추가는 Auditor 사전 승인 필수" 명시. SECURITY.md §8 화이트리스트에 동일 반영.
4. **[후속 관리] CLI 추적 테이블 드리프트**: `20260422000003`(WL-153) + `20260422000004`(WL-154) 모두 SQL Editor 경로로 Cloud만 적용 → `supabase migration list`가 Local-only로 표시. → Cloud 적용 후 `supabase migration repair --status applied 20260422000003 20260422000004` 실행으로 동기화.

### Resolution

**Q1 — 서버 Resolve `partner_id` override 증명**:
- **결정**: 이중 방어 적용.
  - (a) Zod 레이어 — `leadSchema.strict()` 적용 후 unknown key 거부. 단위 테스트 3건(`partner_id`, `status`, `created_at` 각각)으로 실증.
  - (b) JS 레이어 — INSERT payload 순서 `{ ...parsed.data, partner_id, status: 'new' }`로 서버 도출값이 스프레드를 최종 override. 
- **근거**: `tests/lib/schemas/lead.test.ts` 7건 통과. `src/app/[partnerId]/actions/leads.ts:49-53`.

**Q2 — Cloud 적용 전 RED 단언 증명**:
- **결정**: 로컬 Supabase에 `20260422000004_drop_leads_public_insert.sql` 적용 후 `rls-isolation.test.ts §11` `[intended-design / WL-154]` 단언 차단 테스트 GREEN. 정책 제거 전이면 anon이 is_active 파트너 UUID로 INSERT 성공 → 테스트 RED. Red-Green 관계는 정책 상태에 의해 결정됨.
- **근거**: `npx vitest run tests/security/rls-isolation.test.ts` → §11 통과 + §1~§10 regression 0. 로컬 Supabase 적용 후 기준.

### 외부 AI 오진 기록 (Negative Knowledge Base)

- **2차 오진 #1 — Stack Hallucination (Drizzle ORM)**: `db.insert(leads).values(finalPayload)` 예시 제안. 프로젝트는 Supabase JS 클라이언트(`supabaseAdmin.from('leads').insert({...})`) 전용. ORM 도입 없음.
- **2차 오진 #2 — Scope Creep (DEBT-009 포함 요구)**: "DEBT-009를 WL-154에 포함하라. 후속 티켓으로 미루는 것은 잠재적 폭탄 방치"라며 원자성 위반 요구. CLAUDE.md §11.2/§11.4 규정 위반.
- **2차 오진 #3 — Enforcement 오인**: "proxy.ts 외부에서 submitLead가 호출될 수 있다"는 가설. Next.js Server Action은 렌더된 페이지 경로로 POST되어 matcher를 구조적으로 강제로 탄다 — 정책이 아닌 Architecture by Design.
- **3차 오진 — Zod API 오해**: `.strict()` 적용 시 UTM 등 미래 확장 불가 주장. 실제는 `.extend()` / `.merge()`로 명시적 확장 가능. Strict의 회귀 방지 설계 의도와 정확히 부합.
- **학습**: 외부 AI의 "이론적 완벽성" 주장은 (a) 프로젝트 실 스택 무시, (b) 운영 규정 무시, (c) API 문서 부정확 인지 세 가지 패턴이 반복. 모든 권유는 **코드 + 규정 + 실측** 3중 대조 후 수용.
