## Auditor Digest

### Core Logic

`partners.features` JSONB에 쓰기 시점 타입 안전성을 부여하는 이중 방어(Defense in Depth) 도입.
- **Layer 1 (RPC `update_partner_feature`)**: IF p_feature_key NOT IN (...) → RAISE EXCEPTION + `jsonb_build_object` 재구성으로 legacy key 자동 prune (인스턴스 기반 점진적 정제).
- **Layer 2 (DB CHECK `partners_features_shape`)**: IMMUTABLE 함수 `is_valid_partner_features(jsonb)`로 객체 타입 + allowlist key + boolean value 3축 검증. NOT VALID + VALIDATE 2단계로 기존 행 위반 리스크 격리.

### Potential Risks (Auditor sub-agent 발견 6건)

1. **HIGH — Backfill Precondition 미검증**: ADD CONSTRAINT 즉시 적용 시 Cloud 기존 행 중 legacy key 보유 행이 있으면 전체 롤백. Pre-flight 쿼리 + NOT VALID + VALIDATE 2단계로 해결.
2. **HIGH — RPC `||` 병합 Trap**: legacy key 보유 행에서 정상 key 토글 시 CHECK 위반으로 정상 작업까지 영구 차단. `jsonb_build_object` 재구성으로 해결 (허용 key만 유지).
3. **MEDIUM — operator 안정성**: `jsonb - text[]` 연산자와 AND 선단락 비보장 조합. IMMUTABLE 함수 + `jsonb_each` 조합으로 우회.
4. **MEDIUM — 에러 메시지 generic**: CHECK 위반 시 "어느 조건"을 구분 불가. 진단 쿼리를 Digest에 영속화하여 future-self 방어.
5. **MEDIUM — FEATURE_KEYS 4곳 drift**: TypeScript 상수 / CHECK 함수 / RPC IF / 테스트 sentinel — 코드·DB drift 위험. **DEBT-012** 신규 등록 + 행동 기반 drift 탐지 Vitest 테스트.
6. **LOW — CREATE OR REPLACE 순서 의존성**: 20260419000007 RPC 본문을 20260422000002로 치환. 파일 상단 주석으로 순서 명시.

## 진단 쿼리 (future-self 방어 영속화)

### Cloud Pre-flight (Q1용)

```sql
-- partners.features 행 중 신규 CHECK 제약을 위반할 행이 있는지 사전 실증
SELECT id, subdomain, features,
       jsonb_typeof(features) AS type,
       features - ARRAY['custom_domain','analytics','multi_locale']::text[] AS extra_keys
FROM public.partners
WHERE NOT (
  jsonb_typeof(features) = 'object'
  AND (features - ARRAY['custom_domain','analytics','multi_locale']::text[]) = '{}'::jsonb
  AND NOT EXISTS (
    SELECT 1 FROM jsonb_each(features) e
    WHERE jsonb_typeof(e.value) != 'boolean'
  )
);
-- 기대: 0건. 1건 이상 시 Backfill 필수.
```

### 제약 위반 원인 분류 (에러 메시지 generic 보완)

```sql
-- CHECK 제약 위반 시 "어떤 축"이 위반했는지 분해 진단
SELECT id, subdomain, features,
       CASE
         WHEN jsonb_typeof(features) != 'object' THEN 'type_mismatch'
         WHEN (features - ARRAY['custom_domain','analytics','multi_locale']::text[]) != '{}'::jsonb THEN 'unknown_key'
         WHEN EXISTS (SELECT 1 FROM jsonb_each(features) e WHERE jsonb_typeof(e.value) != 'boolean') THEN 'non_boolean_value'
         ELSE 'unknown_cause'
       END AS violation_axis
FROM public.partners
WHERE NOT public.is_valid_partner_features(features);
```

## WL-124 회고 (Q3 결정 — 병기 섹션)

> **배경**: `docs/audits/WL-124.md` 가 누락되어 당시 `features` JSONB 인프라 도입 시 검증 맥락이 세션 종료와 함께 유실됨. WL-150에서 드러난 취약점이 WL-124의 잠재 결함을 사후 노출시켰기에, 본 절에서 WL-124 회고를 함께 기록하여 아키텍처 히스토리 연속성을 복원한다.

### WL-124 원본 스코프 (복원)

- 마이그레이션: `supabase/migrations/20260419000006_partners_add_features.sql` — `features JSONB NOT NULL DEFAULT '{}'::jsonb`
- RPC: `supabase/migrations/20260419000007_update_partner_feature_fn.sql` — `update_partner_feature(p_partner_id, p_feature_key TEXT, p_enabled BOOLEAN)` SECURITY INVOKER
- GIN 인덱스: `supabase/migrations/20260420000001_add_partners_features_gin.sql` — `features` 필터 쿼리 풀스캔 방지
- 앱 레이어: `src/lib/features/features-schema.ts` (Zod) + `src/lib/features/has-feature.ts` (unknown key console.warn + false)
- 트랙: HIGH-Standard (nullable → NOT NULL DEFAULT 카탈로그 레벨 처리, UPDATE 미포함)

### WL-124 당시 Audit에서 놓친 점 (WL-150에서 사후 노출)

1. **"Garbage In" 방어 부재**: 앱 레이어의 관용 처리(`console.warn + false`)만으로 안전하다고 간주. DB 레이어 shape 제약 결여가 "허용 외 구조가 저장될 수 있음"을 용인 상태로 방치. → **WL-150 Layer 2 CHECK 제약 도입**으로 해소.
2. **RPC key 미검증**: `p_feature_key TEXT`를 `jsonb_build_object`로 바로 병합. `update_partner_feature('beta_xyz', true)` 같은 호출이 DB에 임의 key 투입 가능. → **WL-150 Layer 1 IF allowlist**로 해소.
3. **병합 연산자 선택 착오**: `||` 연산자의 "기존 key 보존" 특성이 allowlist 설계와 충돌할 가능성 미인지. legacy key 보유 시 Trap 형성 위험. → **WL-150 jsonb_build_object 재구성**으로 해소.
4. **동기 지점 문서화 누락**: `FEATURE_KEYS` 확장 시 수정해야 할 DB·코드 4곳(Zod 스키마 / CHECK 함수 / RPC IF / 테스트 sentinel)이 문서화되지 않음. → **WL-150 주석 보강 + DEBT-012 등록**으로 부분 해소 (구조적 해결은 DEBT-012의 장기 과제).

### 학습 (Negative Knowledge Base 등재)

- **JSONB 컬럼 도입 시 DB 레이어 shape 제약은 "나중에"가 아니라 "초기"에 설계한다.** WL-124 시점에 CHECK 제약을 함께 도입했다면 WL-150 Auditor 취약점 1·3이 원천 발생 불가능이었음.
- **RPC가 allowlist를 우회하는 설계는 RPC가 아니라 "얇은 SQL 래퍼"다.** `SECURITY INVOKER` 경유라도 allowlist 게이트는 RPC 고유 책임.
- **Auditor Digest 파일 영속화는 High 트랙 Ticket에서도 중요**: WL-124 당시 채팅에만 있던 Digest가 세션 종료 시 유실되어 WL-150에서 맥락 역추적에 시간 소모. CLAUDE.md §5 "Auditor Digest 영속화" 규정의 실천 중요성 재확인.
