# arch-eval — 개발구조 건강도 평가

## 용도

특정 시점 기준으로 코드베이스의 아키텍처 건강도를 **커밋 시점으로 증명된 사실**만으로 냉엄하게 평가한다.
"느낌상 좋아진 것 같다"를 "커밋 해시로 증명된 점수"로 변환하는 것이 목적이다.

## 사용법

```
/arch-eval [평가날짜]
```

예: `/arch-eval 260420`

날짜를 생략하면 오늘 날짜를 기준으로 한다.

---

## 실행 절차

### STEP 1 — 직전 평가 양식 확정

1. Confluence 부모 페이지(`299794481`) 하위 자식 페이지 목록을 `getConfluencePageDescendants`로 조회한다.
2. `개발구조평가-[날짜]` 패턴의 페이지 중 가장 최근 것을 `getConfluencePage(contentFormat: "markdown")`으로 읽는다.
3. 섹션 구조(평가 항목 번호·점수 형식·개선 필요 사항 표 컬럼·종합 점수표·로드맵 형식)를 확정한다.
4. **양식을 확정하기 전까지 평가를 시작하지 않는다.**

### STEP 2 — 실측 데이터 수집

평가 기준 시점의 `git log`와 코드베이스를 실측한다. **느낌이나 추측으로 검증을 대체하지 않는다.**

| 측정 항목 | 명령 |
|----------|------|
| 평가 기준일 최종 커밋 해시 | `git log --oneline --before="[날짜] 23:59" -1` |
| 직전 평가 이후 커밋 목록 | `git log --oneline --after="[직전평가날짜]" --before="[평가날짜] 23:59"` |
| 직전 평가 개선 사항 해결 여부 | `git log --oneline --grep="WL-XXX"` + Grep |
| 기능 구현 존재 여부 | Glob + Grep으로 파일·함수 실존 확인 |
| 테스트 결과 수치 | `git log` 커밋 메시지 내 vitest/playwright 수치 확인 |

**커밋 시점 반드시 확인**: "구현됨"으로 판정하려면 평가 기준일 이전 커밋인지 반드시 확인한다.
평가일 이후 커밋은 해당 시점의 상태가 아니다 — 이전 평가의 WL-42 오진 사례를 참고할 것.

### STEP 3 — 10개 항목 냉엄 평가

직전 평가의 ⚠️ 개선 필요 사항 항목별로 해결 여부를 실증하고 점수를 판정한다.

**평가 원칙**:
- 직전 평가 문제점이 해결되었으면 → 점수 상승. 근거는 커밋 해시로 제시한다.
- 해결되지 않았으면 → 점수 유지 또는 하락. 이유 없는 점수 인상 금지.
- 새로운 문제점 발견 시 → 점수 하락.
- "노력했다", "시도했다", "개선 중이다"는 점수 상승 근거가 아니다. **완료된 커밋**만 인정한다.

| # | 평가 항목 |
|---|----------|
| 1 | 관심사의 분리 (Separation of Concerns) |
| 2 | 타입 안전성 (Type Safety) |
| 3 | 보안 (Security) |
| 4 | 성능 (Performance) |
| 5 | 확장성 (Scalability) |
| 6 | 유지보수성 (Maintainability) |
| 7 | 에러 처리 (Error Handling) |
| 8 | 테스트 가능성 (Testability) |
| 9 | 코드 일관성 (Code Consistency) |
| 10 | 문서화 (Documentation) |

각 항목 작성 형식 (직전 평가 양식을 그대로 계승):
```
## N. [항목명]

**점수: X.X / 5** [↑/→/↓] ([직전평가] 대비 [변화])

> **[장점]**
> - [커밋 해시] 또는 [파일:줄번호] 기반 근거

### ⚠️ 개선 필요 사항

| ID | 위치 | 설명 |
|----|------|------|
| A  | ...  | ...  |
```

### STEP 4 — 종합 점수표 + 최종 판정 + 로드맵 작성

**종합 점수표**: 직전 평가 → 현재 평가 항목별 점수 변화표.

**최종 판정**: 수치 나열이 아닌 한 문장 제목 + 잘한 것 / 문제점 / 핵심 원인 서술.

**향후 과제 로드맵**: 긴급·단기·중기 3단계로 구분.

### STEP 5 — 초안 사용자 확인

작성된 평가 전문(全文)을 사용자에게 보여주고 수정 요청을 받는다.
사용자 확인 없이 바로 Confluence에 발행하지 않는다.

### STEP 6 — Confluence Live Doc 자식 페이지 생성

사용자 확인 후 `createConfluencePage`로 신규 자식 페이지를 생성한다.

| 파라미터 | 값 |
|---------|---|
| cloudId | `opsnowinc.atlassian.net` |
| spaceId | `290848804` (WS Space) |
| parentId | `299794481` (개발구조 평가 부모) |
| title | `개발구조평가-[날짜]` (예: `개발구조평가-260420`) |
| contentFormat | `markdown` |
| subtype | `live` ← **Live Doc 필수. 일반 페이지로 발행 금지.** |

## 공통 실수 방지

| 실수 | 방지책 |
|------|--------|
| 커밋 시점 확인 없이 "구현됨" 판정 | 반드시 `git log --before` 로 기준일 이전 커밋인지 확인 |
| 평가일 이후 커밋을 해당 시점 성과로 인정 | 기준일 이전 커밋만 점수 근거로 인정 |
| 직전 평가 양식을 임의로 변경 | STEP 1 양식 확정 후 그대로 계승 |
| 사용자 확인 없이 바로 Confluence 발행 | STEP 5 확인 루프 필수 |
| "노력/시도"를 점수 근거로 사용 | 완료된 커밋 해시만 인정 |
